# TECH_SPEC — Техническая спецификация

## Общая техническая концепция

ChatForge — multi-tenant SaaS-платформа для создания AI-чатботов.
Система состоит из Elixir-бэкенда (Phoenix), React-фронтенда (TypeScript) и PostgreSQL.

Ключевые принципы:
- Горизонтальное масштабирование через stateless-сервисы.
- Изоляция данных тенантов на уровне доменной модели (tenant_id во всех таблицах).
- Асинхронное взаимодействие через PubSub и Event Bus.
- Минимальные задержки для real-time функций (streaming AI-ответов).
- Конфигурация через переменные окружения — никакого хардкода.

---

## Архитектурный стиль

**Modular Monolith** (Elixir/Phoenix) с чётким разделением на контексты (bounded contexts).

> Почему не микросервисы: на этапе MVP модульный монолит на Elixir даёт все преимущества
> изоляции контекстов без операционной сложности микросервисов. Elixir/OTP нативно
> поддерживает изоляцию процессов, что делает модульный монолит естественным выбором.
> При необходимости контексты легко выносятся в отдельные сервисы.

Коммуникация:
- Sync: HTTP/REST (Phoenix controllers) + JSON API
- Async: Phoenix.PubSub (внутри), Broadway/Oban (фоновые задачи)
- Real-time: Phoenix Channels (WebSocket) для streaming AI-ответов

## AI и модели

- Поддерживаются внешние AI-провайдеры: OpenAI, Anthropic, Google (через конфигурацию).
- Выбор провайдера и модели — через env-переменные, без хардкода.
- Возможность замены провайдера без изменения бизнес-логики.
- Все промпты хранятся в БД и версионируются.
- Контроль токенов и стоимости на уровне AI Orchestrator контекста.
- Streaming ответов через Phoenix Channels (WebSocket → клиент).
- System prompt задаётся Creator-ом при настройке чата (опционально в будущем).

## Безопасность

- Аутентификация через JWT-токены (access + refresh).
- Access token: короткоживущий (15 мин), в Authorization header.
- Refresh token: долгоживущий (30 дней), httpOnly cookie.
- Хеширование паролей: bcrypt.
- TLS для всех внешних соединений.
- Rate limiting на уровне Traefik middleware + Plug (Phoenix).
- Изоляция данных тенантов: tenant_id в каждой таблице, проверка на уровне Ecto queries.
- CORS: настраивается per-tenant (поддомены).
- Аудит всех действий через Event Bus.

## Окружения

### Development
- Docker Compose: PostgreSQL, Redis, MinIO.
- Elixir/Phoenix запускается локально (`mix phx.server`).
- React dev server через Vite (`npm run dev`).
- Mock AI для разработки без реальных провайдеров (опционально).
- Seed-данные для быстрого старта.

### Staging
- Полная копия production.
- Реальные AI-провайдеры с тестовыми ключами.

### Production
- Docker-контейнеры, оркестрация (Kubernetes или Docker Swarm).
- CI/CD pipeline через GitHub Actions.
- Централизованный мониторинг (Prometheus + Grafana).
- Структурированное логирование (JSON).
- Автоматические бэкапы PostgreSQL.

## Ограничения (важно для агента)

- ❌ Не добавлять технологии без явной причины и согласования.
- ❌ Не объединять контексты с разными зонами ответственности.
- ❌ Не писать бизнес-логику во frontend.
- ❌ Не использовать синхронные вызовы там, где нужен PubSub/Oban.
- ❌ Не хардкодить промпты, ключи API, конфигурацию.
- ❌ Не обращаться к AI API напрямую из контроллеров — только через AI контекст.
- ✅ Придерживаться event-driven подхода.
- ✅ Код только production-ready.
- ✅ Все запросы к БД фильтруются по tenant_id.

## Итог

TECH_SPEC определяет **КАК** реализуется система.
Elixir/Phoenix для бэкенда, React/TypeScript для фронтенда, PostgreSQL как основное хранилище.
Модульный монолит с чёткими bounded contexts, готовый к декомпозиции при масштабировании.
