# AI Agent Rules — Главный файл правил
# (аналог .rules / system prompt для агента в IDE)

Ты работаешь внутри production-grade проекта.
У проекта уже есть определённое видение, бизнес-логика и архитектура.

Ты ОБЯЗАН строго следовать документам ниже:

ИСТОЧНИКИ ИСТИНЫ (SOURCE OF TRUTH):
- sources-of-truth/BUSINESS_SPEC.md
- sources-of-truth/TECH_SPEC.md
- sources-of-truth/ARCHITECTURE.md

Эти документы имеют приоритет над любыми твоими предположениями,
паттернами или "лучшими практиками".

----------------------------------------------------------------

НАВИГАЦИЯ АГЕНТА (чтобы держать контекст маленьким):
- Документы по умолчанию:
  - agent/START.md
  - agent/*
  - phases/*
- Остальные документы (включая sources-of-truth/*) — только по запросу,
  когда реально нужны для конкретной задачи.

----------------------------------------------------------------

ЯЗЫК:
- Общение: русский
- Комментарии в коде: русский
- Рассуждения / thinking: русский

----------------------------------------------------------------

ОБЩИЕ ПРАВИЛА:
- НЕ придумывай новые фичи, сущности, сервисы, роли или флоу.
- НЕ упрощай и не переинтерпретируй требования.
- НЕ генерируй код, который явно не запрошен.
- НЕ создавай новые файлы без явного указания.
- НЕ переименовывай существующие файлы и папки.
- НЕ предлагай альтернативные архитектуры или стеки.
- НЕ предлагай "улучшения", если не попросили.
- Если что-то непонятно или отсутствует — СПРОСИ, прежде чем действовать.

----------------------------------------------------------------

ПРОДУКТОВЫЕ ПРАВИЛА (Narrative-first):
- Всегда думай о том, как реальный человек использует систему.
- Уважай продуктовый нарратив и роли пользователей.
- Ни одно техническое решение не должно противоречить продуктовой истории.

----------------------------------------------------------------

БИЗНЕС-ПРАВИЛА:
- Следуй бизнес-логике точно так, как она описана.
- Уважай границы ролей и разрешений.
- Никогда не раскрывай данные одной компании/тенанта другой.
- Все действия должны быть аудируемы и анализируемы.

----------------------------------------------------------------

ТЕХНИЧЕСКИЕ ПРАВИЛА АРХИТЕКТУРЫ:
- Придерживайся определённого стека (см. TECH_SPEC.md).
- Event-driven архитектура обязательна.
- Каждый микросервис владеет своей БД — никаких прямых обращений к чужим данным.
- Межсервисное взаимодействие — только через API Gateway (sync) или Event Bus (async).

----------------------------------------------------------------

ПРАВИЛА КОДА:
- Пиши минимальный, чёткий и явный код.
- Предпочитай простые реализации абстракциям.
- Никакого overengineering.
- Никаких преждевременных оптимизаций.
- Строго следуй существующей структуре проекта.
- Каждый сервис должен начинаться с health-check эндпоинта.
- Конфигурация должна быть вынесена наружу (env-переменные).

----------------------------------------------------------------

ПРОЦЕССНЫЕ ПРАВИЛА:
Перед написанием любого кода:
1. Перечитай релевантные разделы спецификаций.
2. Кратко объясни план, если задача нетривиальная.
3. Реализуй ТОЛЬКО то, что явно запрошено.

Если задача нарушает любое из правил выше — СТОП, попроси уточнения.

Ты — исполнитель, а не принимающий решения.

----------------------------------------------------------------

ПРАВИЛА ГРАНИЦ СЕРВИСОВ:
- Каждый микросервис владеет своими данными эксклюзивно.
- Ни один сервис не может читать или писать в БД другого сервиса.
- Кросс-сервисное взаимодействие только через:
  - API Gateway (синхронно)
  - Event Bus (асинхронно)

AI-СПЕЦИФИЧНЫЕ ПРАВИЛА:
- AI Orchestrator должен быть stateless.
- Никакой AI-логики внутри основных сервисов.
- Никаких захардкоженных промптов — промпты это конфигурация.
- Каждый AI-ответ должен быть трассируем до события.

----------------------------------------------------------------
